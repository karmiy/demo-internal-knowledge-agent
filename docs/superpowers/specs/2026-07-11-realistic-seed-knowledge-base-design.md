# 真实感预置知识库数据设计

## 目标

将当前只有 3 份短文档的骨架 Demo 扩充成一个适合问答、权限和文档预览测试的小型 SaaS 公司内部知识库。管理员应能在现有“文档管理”页面看到并预览全部预置文档；不新增工资表页面、员工管理页面或其他业务功能。

## 公司背景

数据模拟一家约 30 人的 SaaS 软件公司。现有三个演示身份保持不变：

- `alice.programmer`：Engineering 部门，programmer 角色；
- `helen.hr`：People 部门，hr 角色；
- `andy.admin`：Operations 部门，admin 角色。

工资数据继续作为 PostgreSQL 结构化数据保留，仍通过受控 Agent 工具查询，不作为知识库文档展示。

## 文档集合

知识库固定包含 12 份 Markdown 文档，每份包含 4–6 个有实际可问事实的章节。

### 所有已登录员工可见

1. **员工手册**（`employee-handbook.md`）：工作时间、行为准则、沟通原则、办公资源、离职交接。
2. **考勤与休假制度**（`attendance-leave-policy.md`）：打卡异常、年假、病假、事假、调休及审批时限。
3. **差旅与报销制度**（`travel-expense-policy.md`）：出差申请、交通住宿标准、票据要求、提交时限及审批链路。
4. **信息安全规范**（`information-security-policy.md`）：账号与 MFA、数据分级、设备安全、外发规则及安全事件上报。
5. **远程办公指南**（`remote-work-guide.md`）：适用范围、协作时间、会议规范、办公环境及信息安全要求。
6. **新员工入职指南**（`onboarding-guide.md`）：入职首日、首周任务、导师机制、试用期目标及转正流程。

### Engineering 部门可见

7. **工程研发规范**（`engineering-guide.md`）：分支策略、代码评审、自动化测试、依赖管理及技术文档。沿用现有 seed 文件名以保证数据卷原地升级。
8. **发布与变更管理**（`release-change-management.md`）：变更分级、发布窗口、审批、上线验证、回滚及复盘。
9. **故障响应与值班手册**（`incident-response-oncall.md`）：P1/P2/P3 分级、响应时限、升级路径、沟通频率及事后复盘。

### HR 与 Admin 可见

10. **薪酬与职级制度**（`hr-compensation-policy.md`）：职级框架、薪酬构成、年度复核、调薪触发条件及保密要求；不包含个人工资明细。沿用现有 seed 文件名以保证数据卷原地升级。
11. **绩效评估制度**（`performance-review-policy.md`）：目标设定、季度回顾、年度评估、评级校准、申诉及绩效改进计划。

### Admin 可见

12. **采购与供应商管理制度**（`procurement-vendor-policy.md`）：采购分级、比价、合同审批、供应商准入、验收及续约评估。

## 内容原则

- 每个章节提供明确、可验证的事实，例如天数、时间段、金额档位、审批角色或响应时限。
- 内容为虚构 Demo 数据，不使用真实公司、员工、客户、供应商或财务信息。
- 主题之间避免大段重复，减少本地词法 Embedding 的检索噪声。
- 标题和章节使用用户自然提问时可能采用的词汇。
- 每份文档保持中等长度，便于抽屉预览，不生成冗长填充文本。
- 核心验收事实固定为：核心协作时间 `10:00–16:00`；普通年假至少提前 3 个工作日申请；差旅报销在返程后 10 个工作日内提交；P1 故障 10 分钟内响应且每 30 分钟同步进展；生产发布至少需要代码所有者批准并验证健康检查；年度薪酬复核在每年 3 月启动；单次采购达到 `50,000 CNY` 时至少取得 3 家供应商报价。

## 权限预期

- `alice.programmer` 可检索 6 份全员文档和 3 份 Engineering 文档，共 9 份。
- `helen.hr` 可检索 6 份全员文档和 2 份 HR 文档，共 8 份。
- `andy.admin` 通过现有管理员绕过规则访问全部 12 份文档，并在前端文档管理页看到、点击和预览全部文档。
- 普通用户仍看不到文档管理导航和管理员文档详情 API。

## 已有数据卷升级

当前 seed 仅按 checksum 查找文档。若修改原有文档内容，checksum 会变化，但 `source_path` 唯一约束不变，直接新增会发生冲突。因此 seed 必须改为以稳定的目标 `source_path` 作为预置文档身份：

1. 新文件不存在时创建 Document、权限和 PENDING 状态；
2. 文件存在且 checksum 未变化时保持现有 READY 文档和 chunks；
3. 文件存在但内容变化时原地更新标题、checksum 和目标文件，将状态设为 PENDING，由 Worker 重新 Ingest；
4. 权限集合以 seed 配置为准进行同步，删除已经不需要的旧权限并补充缺失权限；
5. 重复执行 seed 不创建重复 Document，也不反复重建未变化文档。

不删除数据库卷，不要求用户执行 `docker compose down -v`。

## README 与测试问题

README 更新预置数据说明、权限数量和一组覆盖不同主题的建议问题。至少包含：

- 年假申请需要提前多久？
- 差旅报销应在返程后多久提交？
- P1 故障需要多快响应？
- 生产发布需要哪些审批和验证？
- 薪酬复核通常在什么时候进行？
- 采购达到什么条件需要多家比价？

## 验收标准

- 新安装和保留现有 Docker volume 两种场景下，数据库都恰好有 12 份 seed 管理的预置文档且无重复 source path；用户自行上传的其他文档不计入该数量。
- 12 份文档全部进入 READY，所有 chunks 使用 `local-hash-v2`。
- `andy.admin` 的文档列表返回 12 份并可读取任意一份详情。
- Alice、Helen 和 Andy 的检索结果符合 9、8、12 份权限边界。
- 至少对全员、Engineering、HR 和 Admin 四类文档各执行一个真实问答，回答包含预期事实和正确引用。
- 后端、前端测试和生产构建继续通过。
